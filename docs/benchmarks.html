Key Takeaways

  • Zero overhead for reads. Across every SELECT scenario, bpdbi_raw lands within ±1% of raw JDBC. The binary protocol and lazy decoding add no measurable cost — both drivers are equally network-bound.
  • Pipelining dominates batch inserts: +57–65% vs JDBC batch. executeMany sends Parse once plus N Bind/Execute messages in a single TCP write, amortizing latency the same way reWriteBatchedInserts does for Spring and JDBI.
  • Record mapper is nearly free (~1–2% overhead); bean mapper costs ~8–9%. The record mapper's constructor-based instantiation is cheaper than the bean mapper's reflection-based setter calls. Both are far ahead of Hibernate's entity lifecycle.
  • Kotlin @Serializable mapper is the fastest single-row path at ~411 ops/s, beating JDBI's Kotlin reflection mapper by 21%. Compile-time code generation eliminates runtime reflection entirely.
  • Hibernate is consistently 40–53% behind on reads, 95–97% on writes. Entity lifecycle management, dirty checking, and per-statement round-trips make it the slowest option in every scenario. jOOQ and Sql2o suffer similar batch-insert penalties.
